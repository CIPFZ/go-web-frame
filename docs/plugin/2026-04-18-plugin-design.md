# Plugin 模块设计方案

日期：2026-04-18  
分支：`codex/plugin-module`

## 1. 背景与目标

`plugin` 模块的目标，是在现有系统中新增一套完整的插件中心能力，覆盖插件项目接入、版本提交流转、审核工单池处理、版本上下架以及公开展示。

这套能力要解决的不是“插件信息录入”本身，而是以下几个核心问题：

1. 插件提供方如何持续管理自己的插件项目和版本。
2. 插件版本如何进入标准化审核流，而不是通过线下口头协作。
3. 审核员如何通过公共工单池协同处理，而不是被动等待分派。
4. 管理员如何对卡住的审核流程进行干预。
5. 通过审核的插件如何形成可对外查询的插件列表与插件详情。

## 2. 一期范围

一期范围包含：

1. 后台插件项目管理。
2. 插件版本创建、编辑、提交审核、打回后重提。
3. 工单池查询、认领、审核通过、审核打回。
4. 已发布版本的下架申请与下架审核。
5. 兼容产品信息维护与版本适配关系维护。
6. 发布流转事件记录与时间轴展示。
7. 公开插件列表与插件详情查询接口及页面。

一期明确不做：

1. 通知中心联动。
2. AI 自动扫描结果接入。
3. 批量认领、批量审核。
4. 独立归档中心。
5. 复杂版本 diff 对比。

## 3. 角色与职责

### 3.1 插件提供方

角色 ID：`10010`

职责：

1. 创建和维护插件项目。
2. 创建和维护插件版本。
3. 上传测试报告、插件包、变更说明。
4. 提交审核。
5. 接收打回结果并修改后重新提交。
6. 对已发布版本发起下架申请。

### 3.2 插件审核员

角色 ID：`10013`

职责：

1. 查看工单池中的待处理工单。
2. 主动认领工单。
3. 审核版本内容、适配信息和交付资料。
4. 执行通过、打回、发布、下架等动作。

### 3.3 管理员

角色 ID：`1`

职责：

1. 具备审核员全部能力。
2. 可查看全量插件项目和发布单。
3. 可强制重置处理中工单。
4. 可维护产品字典和关键系统配置。

## 4. 核心设计原则

### 4.1 工单池认领模式

所有提交审核的发布单统一进入公共工单池，不再采用“提交时指定审核人”的模式。

规则如下：

1. 提交审核时不指定审核员。
2. 审核员从工单池主动认领。
3. 一旦认领成功，该工单只能由认领人继续审核。
4. 管理员可对异常占用的工单强制重置。

这个模式的好处是：

1. 审核任务分配更灵活。
2. 避免审核人离岗造成单点阻塞。
3. 审核员工作量更容易按实时情况平衡。

### 4.2 生命周期状态与处理状态分离

发布单同时维护两套状态：

1. `status`：描述发布单的生命周期状态。
2. `process_status`：描述工单在审核池中的处理状态。

之所以分离，是因为“版本是否已通过审核”和“工单当前是否被认领处理中”是两件不同的事情，混在一个字段里会导致语义混乱。

### 4.3 发布单是审核流的核心对象

插件项目是长期主体，但真正进入审核流的是“某个插件项目下的某个发布单”。  
因此所有审核动作、事件流和工单池处理，均围绕发布单展开。

## 5. 业务对象模型

### 5.1 插件项目 Plugin

表示插件主体，包含：

1. 插件编码。
2. 中英文名称。
3. 仓库地址。
4. 中英文描述。
5. 所属部门。
6. 负责人。
7. 创建人。

一个插件项目可以存在多个发布单。

### 5.2 发布单 Release

表示某次插件版本发布或下架申请，是流程处理主对象，包含：

1. 所属插件项目。
2. 请求类型。
3. 生命周期状态 `status`。
4. 工单处理状态 `process_status`。
5. 版本号。
6. 认领人和认领时间。
7. 自检信息、测试报告、x86/ARM 包地址。
8. 变更说明、审核意见、下架原因。
9. 提交、审核通过、发布、下架等关键时间。

### 5.3 兼容产品 Compatible Product

用于描述某个发布单适配哪些产品及其版本约束。

一个发布单可以适配多个产品，每个产品可以有独立的版本约束，例如：

1. `HCI >= 5.2`
2. `aCLI >= 2.1`

### 5.4 发布事件 Release Event

用于记录发布单全流程操作轨迹，包括：

1. 创建。
2. 提交审核。
3. 认领。
4. 审核通过。
5. 审核打回。
6. 重新提交。
7. 发起下架。
8. 执行发布或下架。
9. 管理员强制重置。

事件流既用于审计，也用于前端时间轴展示。

### 5.5 产品字典 Product

维护插件可适配的产品集合，是兼容关系的基础字典数据。

## 6. 状态机设计

### 6.1 发布单生命周期状态 `status`

| 值 | 含义 |
| --- | --- |
| `0` | 草稿 |
| `1` | 已补充资料，待提交审核 |
| `2` | 待审核 |
| `3` | 审核通过，待执行发布或下架 |
| `4` | 已打回 |
| `5` | 已发布 |
| `6` | 已下架 |

### 6.2 工单处理状态 `process_status`

| 值 | 含义 |
| --- | --- |
| `0` | 待处理 |
| `1` | 处理中 |
| `2` | 已退回 |
| `3` | 已完成 |

### 6.3 关键状态流转

正常发版流程：

1. 草稿 `0`
2. 补充资料 `1`
3. 提交审核 `2`
4. 审核通过 `3`
5. 发布完成 `5`

打回重提流程：

1. 待审核 `2`
2. 审核打回 `4`
3. 提供方修改后重新提交 `2`

下架流程：

1. 已发布 `5`
2. 发起下架申请并进入待审核 `2`
3. 审核通过后执行下架 `6`

工单池处理流程：

1. 提交审核后进入待处理 `0`
2. 审核员认领后进入处理中 `1`
3. 打回则进入已退回 `2`
4. 发布完成或下架完成后进入已完成 `3`

## 7. 关键业务规则

### 7.1 提交审核规则

1. 只有提供方本人或管理员可以提交。
2. 提交时不得指定审核员。
3. 提交后发布单进入工单池。
4. 提交动作必须记录事件。

### 7.2 认领规则

1. 只有审核员或管理员可以认领。
2. 只有 `process_status=0` 的工单允许认领。
3. 认领必须原子化，避免并发下被多人同时认领。
4. 认领成功后记录 `claimer_id` 和 `claimed_at`。

### 7.3 审核规则

1. 只有认领人本人或管理员可以执行审核动作。
2. 审核通过和审核打回都必须落事件记录。
3. 打回必须填写审核意见。

### 7.4 重新提交流程

1. 只有被打回的发布单允许重新提交。
2. 重新提交后清空认领人。
3. 工单重新回到待处理池中，允许任意审核员重新认领。

### 7.5 强制重置规则

1. 只有管理员可执行。
2. 只有处理中工单允许重置。
3. 重置必须填写原因。
4. 重置后工单回到待处理状态，并清空认领人。

### 7.6 公开展示规则

1. 只有已发布且未下架的版本允许对外展示。
2. 对外接口不暴露内部审核信息。
3. 对外页面展示的插件详情应聚合项目基础信息和当前有效发布版本信息。

## 9. 后端模块设计

建议在 `backend/internal/modules/plugin` 下新增独立模块，结构与现有模块保持一致：

1. `api`
2. `dto`
3. `model`
4. `repository`
5. `service`
6. `router`

### 9.1 模块拆分建议

#### plugin/project

负责插件项目的增改查、项目详情聚合。

#### plugin/release

负责发布单创建、编辑、状态流转。

#### plugin/workorder

负责工单池视图、认领、重置等聚合能力。

#### plugin/product

负责产品字典查询和后台维护。

#### plugin/public

负责公开插件列表和公开插件详情查询。

### 9.2 服务层职责

服务层应承担：

1. 权限校验。
2. 状态流转校验。
3. 事务控制。
4. 工单认领并发保护。
5. 事件记录写入。
6. 聚合查询组装。

Repository 层只做数据读写，不负责业务决策。

## 12. 权限设计

建议从两个层面控制：

### 12.1 页面级

1. 提供方看项目管理和项目详情。
2. 审核员看工单池和项目详情中的审核视图。
3. 管理员看全部页面。

### 12.2 动作级

1. `submit_review` 仅提供方/管理员。
2. `claim` 仅审核员/管理员。
3. `approve/reject/release/offline` 仅认领人/管理员。
4. `reset` 仅管理员。

## 13. 并发与一致性设计

这是该模块的高风险点，必须在设计阶段明确。

### 13.1 工单认领并发

风险：两个审核员同时点击认领。

要求：

1. 认领必须基于当前 `process_status=0` 的条件更新。
2. 更新结果为 0 行时返回“工单已被他人认领”。
3. 认领和事件写入必须在同一事务中完成。

### 13.2 状态流转幂等

风险：前端重复提交导致状态跳变异常。

要求：

1. 每个动作都校验当前状态是否合法。
2. 非法状态直接拒绝。
3. 不允许跨状态跳转。

### 13.3 数据可见性

风险：公开侧读取到未发布或已下架数据。

要求：

1. 公开接口必须只查询发布态有效数据。
2. 后台接口和公开接口必须分开实现，不共享未经筛选的查询结果。

## 14. 测试设计

一期必须覆盖测试，至少包括以下层级。

### 14.1 后端单元测试

覆盖重点：

1. 状态流转合法性校验。
2. 工单认领规则。
3. 管理员重置规则。
4. 重新提交后清空认领人。
5. 公开接口只返回有效发布数据。

### 14.2 后端集成测试

覆盖重点：

1. 发布单创建到发布完成的完整链路。
2. 打回后重新提交链路。
3. 下架申请到下架完成链路。
4. 并发认领场景。

### 14.3 前端测试

覆盖重点：

1. 路由与页面挂载。
2. 工单池列表与状态标签渲染。
3. 认领、打回、重置按钮显示逻辑。
4. 动态路由组件映射是否正确。

### 14.4 验收测试场景

最少验证以下真实流程：

1. 提供方创建项目并提交一个版本，审核员认领后发布成功。
2. 审核员打回版本，提供方修改后重新提交，另一名审核员可重新认领。
3. 审核员认领后不处理，管理员可重置工单。
4. 已发布插件能在公开列表看到，已下架插件在公开列表消失。

## 15. 与当前仓库的差距评估

当前仓库状态：

1. 后端尚无 `plugin` 模块。
2. 前端 `src/pages/plugin/project-center` 目录存在，但实际为空，不能视为已实现能力。
3. 路由、组件映射、服务层、权限点、数据库迁移均尚未落地。

因此 `plugin` 是一个新增业务模块，不是简单补几个页面。

## 16. 推荐实施顺序

建议按以下顺序推进实现：

1. 数据模型和迁移脚本。
2. 后端 `model/repository/service/api/router` 骨架。
3. 项目和发布单基础接口。
4. 工单池认领与重置。
5. 前端项目管理与项目详情。
6. 前端工单池。
7. 公开插件页面。
8. 单元测试、集成测试、联调验证。

## 17. 风险与取舍

### 17.1 最大风险

1. 工单认领并发冲突。
2. 状态机定义不清导致后续逻辑散乱。
3. 提供方视角、审核员视角、公开视角耦合过深。

### 17.2 设计取舍

1. 一期采用单一发布单模型承载发版与下架申请，不额外拆审批单表，降低复杂度。
2. 一期保留产品字典与兼容关系，但不引入更复杂的产品版本表。
3. 一期保留事件表做审计，不先做通知中心，避免首期实现面过大。

## 18. 结论

`plugin` 模块应被定义为一个新的完整业务域。  
它的核心不是插件信息录入，而是“插件项目 + 发布单 + 工单池认领审核 + 对外发布”的闭环流程。

后续实现时，必须优先保证三件事：

1. 状态机清晰。
2. 工单认领并发安全。
3. 前后台视图边界清晰。
