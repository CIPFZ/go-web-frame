# Plugin 模块真实环境测试方案

日期：2026-04-18
分支：`codex/plugin-module`
环境：Docker 实际运行环境

## 1. 测试目标

验证 plugin 模块在真实部署环境中的以下方面：

1. 基础环境可用：前端、后端、PostgreSQL、Redis、MinIO 容器健康。
2. 权限边界正确：管理员、插件提供方、插件审核员的接口权限和页面入口符合设计。
3. 主流程闭环可用：项目创建、发布单创建、提交审核、认领、打回、重新提交、审核通过、执行发布、发起下架、审核下架、执行下架。
4. 工单池机制有效：工单必须先认领后审核；重复认领会被拒绝。
5. 公开展示正确：已发布插件会出现在公开列表和公开详情；下架后不再对外展示。
6. 前端页面可用：动态路由页面能够真实渲染，公开静态路由可访问，页面数据与后端状态一致。

## 4. 测试方法

### 4.1 接口层验证

使用真实 `backend:8080` 接口，携带真实 JWT 或 Cookie，检查：

1. HTTP 状态码
2. 业务 `code/msg`
3. 返回数据字段
4. 状态流转前后的一致性
5. 公开接口结果变化

### 4.2 浏览器端验证

使用真实 `frontend:80` 页面，检查：

1. 登录成功后页面能正常渲染
2. 动态路由页面不出现组件映射错误
3. 列表、详情、公开页能看到接口层创建的真实数据

## 5. 详细用例

### A. 环境健康检查

1. Docker 容器状态全部为 `Up`，数据库容器健康。
2. 前端首页可访问。
3. 后端登录接口可响应。

### B. 账号与权限准备

1. 管理员登录成功。
2. 管理员创建提供方账号成功。
3. 管理员创建审核员 A 账号成功。
4. 管理员创建审核员 B 账号成功。
5. 提供方登录成功。
6. 审核员 A/B 登录成功。

### C. 提供方主流程

1. 提供方创建插件项目成功。
2. 提供方查询项目列表，能看到新项目。
3. 提供方创建版本发布单成功。
4. 提供方查看项目详情，能看到发布单。
5. 提供方提交审核成功。

### D. 审核流与工单池

1. 审核员 A 在工单池看到待处理工单。
2. 审核员 A 认领工单成功。
3. 审核员 B 再次认领同一工单失败。
4. 审核员 A 打回工单成功。
5. 提供方重新提交成功。
6. 审核员 A 再次认领成功。
7. 审核员 A 审核通过成功。
8. 审核员 A 执行发布成功。

### E. 公开展示验证

1. 公开插件列表出现该插件。
2. 公开插件详情可访问，并显示版本和兼容信息。

### F. 下架流程验证

1. 提供方对已发布版本发起下架申请成功。
2. 审核员 A 在工单池再次看到该下架工单。
3. 审核员 A 认领下架工单成功。
4. 审核员 A 审核通过下架申请成功。
5. 审核员 A 执行下架成功。
6. 公开插件列表中不再出现该插件。

### G. 浏览器页面验证

1. 提供方登录后打开项目管理页，页面可渲染且能看到测试项目。
2. 提供方打开项目详情页，页面可渲染且能看到发布单数据。
3. 审核员登录后打开工单池页，页面可渲染。
4. 公开列表页可渲染。
5. 公开详情页在发布后可渲染。
6. 全过程不出现 `未找到组件映射`、空白页、明显前端报错。

## 6. 成功判定

满足以下条件即判定本轮真实环境测试通过：

1. 所有核心状态流转接口返回成功，且状态变化与设计一致。
2. 工单认领互斥生效。
3. 发布前后、下架前后，公开接口返回结果符合预期。
4. 管理端与公开端关键页面均可正常渲染并展示真实数据。
