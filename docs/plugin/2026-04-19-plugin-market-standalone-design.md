# 独立插件生态中心设计方案

## 1. 背景与目标

当前仓库中的插件生态中心已经具备公开浏览和下载能力，但它仍然内嵌在 CMS 前后端内部：

- 前端公开页面仍依赖 CMS 主应用的运行时、路由、构建和鉴权豁免逻辑。
- 后端公开接口仍与 `plugin` 模块的项目管理、工单、审核流程、主数据管理共用同一套实现。
- 部署时只能随 CMS 一起发布，不能单独演进。

本次目标是在同一仓库内新增一套完全独立的“插件生态中心”前后端实现，同时满足以下约束：

1. `master` 分支中现有集成版插件生态中心保留，不删除、不替换。
2. 新生态中心前端仅服务用户浏览、查看详情和下载，不包含 CMS 管理能力。
3. 新生态中心后端仅保留公开市场所需能力，不复用 CMS 插件模块实现。
4. 新生态中心后端使用 `sqlite3`，部署简单、便于快速落地。
5. 新生态中心的数据由 CMS 后续通过同步接口推送，不直接依赖 CMS 数据库。
6. 新生态中心只存储公开展示所需插件数据，不保留事件流、工单、审核轨迹等内部数据。

## 2. 范围

### 2.1 本次设计覆盖

- 独立插件生态中心前端应用的目录结构、路由边界、页面范围和构建方式。
- 独立插件生态中心后端服务的启动入口、模块边界、接口设计和数据库模型。
- CMS 向独立生态中心同步公开数据的协议设计。
- SQLite 数据模型、同步策略、基础鉴权策略和测试策略。
- 与现有 CMS 集成版插件生态中心的并存方式。

### 2.2 本次设计不覆盖

- CMS 侧同步任务的具体实现代码。
- 插件包文件的对象存储迁移。
- 市场服务的运营后台、人工编辑后台或审计后台。
- 插件评分、评论、收藏、搜索推荐等二期能力。

## 3. 总体方案

采用“仓库内双系统并存”的方式：

- 保留当前 CMS 集成版插件生态中心原样存在。
- 新增一个独立前端应用，作为插件生态中心站点。
- 新增一个独立后端服务，作为插件生态中心 API 服务。
- 新增一个独立领域模块 `plugin_market`，不再复用现有 `plugin` 模块实现。
- 独立后端使用 SQLite 持久化公开插件数据。
- CMS 在插件发布、更新、下架时，通过专用同步接口把公开数据推送到市场服务。

这意味着“公开市场”从现在开始是一个独立产品，而不是 CMS 的一组公开页面。

## 4. 目录与模块拆分

### 4.1 前端拆分

新增独立前端应用目录，例如：

```text
plugin-market-front-end/
  config/
  src/
    pages/
      plugin-list/
      plugin-detail/
    services/
    locales/
    app.tsx
  Dockerfile
  package.json
```

设计原则：

- 不依赖当前 CMS 前端的 `app.tsx` 登录态逻辑。
- 不接入动态菜单、权限路由、用户信息初始化。
- 仅保留市场站点所需的国际化、请求封装、页面组件和主题配置。
- 技术栈继续沿用 React + Umi + Ant Design，减少迁移成本。

### 4.2 后端拆分

新增独立后端服务入口，例如：

```text
backend/cmd/plugin-market/
```

新增独立模块，例如：

```text
backend/internal/modules/plugin_market/
  api/
  dto/
  model/
  repository/
  router/
  service/
```

设计原则：

- `plugin_market` 与现有 `plugin` 模块逻辑隔离。
- 市场服务不依赖工单、审核、认领、事件流等 CMS 领域对象。
- 市场服务只聚焦“公开插件目录”和“版本下载能力”。

## 5. 独立前端设计

### 5.1 页面范围

独立前端只保留两类页面：

1. 插件列表页
2. 插件详情页

不包含：

- 登录页
- 用户中心
- CMS 菜单
- 插件项目管理
- 工单池
- 主数据管理

### 5.2 路由设计

建议采用最小路由集：

- `/`
- `/plugins`
- `/plugins/:id`

其中：

- `/` 重定向到 `/plugins`
- `/plugins` 展示插件列表
- `/plugins/:id` 展示插件详情

### 5.3 前端数据依赖

独立前端仅依赖市场服务公开接口：

- 获取插件列表
- 获取插件详情

不再直接调用 CMS 中的 `/api/v1/plugin/public/*`

### 5.4 UI 与体验要求

- 延续当前插件生态中心已有的公开站点视觉风格。
- 保留中英文切换。
- 保留用户浏览、查看版本、下载架构包等核心交互。
- 不出现 CMS 左侧菜单或后台痕迹。

## 6. 独立后端设计

### 6.1 服务职责

市场服务只承担两类职责：

1. 对外提供公开查询接口
2. 接收 CMS 推送的公开插件数据并持久化

它不承担：

- 插件项目创建
- 发布单审核
- 工单池处理
- 审批流转
- 内部事件记录

### 6.2 启动入口

新增独立启动入口：

- `backend/cmd/plugin-market/main.go`

新增独立配置文件：

- `backend/configs/plugin-market.yaml`

配置项至少包括：

- 服务监听地址
- SQLite 文件路径
- CORS
- 日志
- 同步密钥或同步令牌

### 6.3 路由边界

建议将公开接口与同步接口分组：

- `GET/POST /api/v1/market/plugins`
- `GET/POST /api/v1/market/plugins/:id`
- `POST /api/v1/market/sync/plugin/upsert`
- `POST /api/v1/market/sync/plugin/version/upsert`
- `POST /api/v1/market/sync/plugin/version/offline`
- `POST /api/v1/market/sync/plugin/delete`

说明：

- 公开查询接口用于独立前端。
- 同步接口只对 CMS 开放，不对普通用户开放。

## 7. 数据模型设计

### 7.1 建模原则

市场服务只存“公开态最终结果”，不存 CMS 内部过程数据。

因此数据模型要做裁剪：

- 保留插件基础信息
- 保留公开版本信息
- 保留兼容性与下载地址
- 不保留事件流
- 不保留工单状态
- 不保留审核人、认领人、评论轨迹

### 7.2 建议表结构

#### `market_plugins`

字段建议：

- `id`
- `plugin_id`
- `code`
- `name_zh`
- `name_en`
- `description_zh`
- `description_en`
- `capability_zh`
- `capability_en`
- `owner_name`
- `status`
- `latest_version`
- `created_at`
- `updated_at`

说明：

- `plugin_id` 用于保存 CMS 原插件 ID，便于同步幂等。
- `id` 作为市场服务自己的主键。

#### `market_plugin_versions`

字段建议：

- `id`
- `plugin_id`
- `release_id`
- `version`
- `changelog_zh`
- `changelog_en`
- `test_report_url`
- `package_x86_url`
- `package_arm_url`
- `released_at`
- `version_constraint`
- `publisher`
- `status`
- `created_at`
- `updated_at`

说明：

- `release_id` 对应 CMS 发布单 ID，用于同步幂等。
- `status` 只保留市场态，例如 `published` / `offlined`。

#### `market_plugin_compatibilities`

字段建议：

- `id`
- `plugin_version_id`
- `target_type`
- `product_code`
- `product_name`
- `version_constraint`
- `created_at`
- `updated_at`

说明：

- `target_type` 取值例如 `product` / `acli`
- 不再保留 CMS 内部产品主数据的复杂关联，只保留展示必要信息。

## 8. CMS 到市场服务的同步协议

### 8.1 同步方式

采用“CMS 主动推送，市场服务被动接收”的方式。

优点：

- 市场服务不依赖 CMS 数据库。
- 市场服务部署位置独立，不需要访问 CMS 内网数据库。
- 未来可以按环境分别部署，只需打通 API。

### 8.2 同步粒度

建议拆成以下同步动作：

1. 插件基础信息同步
2. 发布版本上架同步
3. 发布版本信息更新同步
4. 发布版本下架同步
5. 插件删除或隐藏同步

### 8.3 幂等要求

所有同步接口必须按以下字段幂等：

- 插件：`plugin_id`
- 版本：`release_id`

同一条同步消息重复发送时，结果必须一致。

### 8.4 鉴权设计

同步接口不走用户态 JWT，而采用服务间固定密钥认证，例如：

- Header：`X-Market-Sync-Token`

市场服务校验：

- token 是否存在
- token 是否与配置一致

这是当前阶段最简单可控的方式，足够满足仓库内双服务部署。

## 9. 查询接口设计

### 9.1 列表接口

返回内容只包含列表页需要的数据：

- 插件 ID
- code
- 中英文名称
- 中英文描述
- 最新版本号
- 最新发布时间
- 下载架构信息
- 兼容信息摘要

### 9.2 详情接口

返回内容只包含详情页需要的数据：

- 插件基础信息
- 所有公开版本
- 当前默认版本
- 每个版本的下载链接
- 兼容信息
- 测试报告链接

不返回：

- 审核状态
- 工单状态
- 事件轨迹
- CMS 内部 ID 之外的敏感管理信息

## 10. 与现有 CMS 集成版的并存策略

本次拆分必须满足“保留原系统”的要求，因此采用并存策略：

1. CMS 现有公开插件市场页面继续保留。
2. CMS 现有 `/api/v1/plugin/public/*` 接口继续保留。
3. 新独立市场前端和后端作为新增能力进入仓库。
4. 原集成版只作为现有能力保留，不作为未来主交付路径。

这样可以保证：

- 旧功能不受影响
- 新功能可逐步接管
- 回滚成本低

## 11. Docker 与部署设计

### 11.1 前端

独立市场前端应有独立 Dockerfile，单独构建和部署。

例如：

- 容器名：`plugin-market-frontend`
- 对外端口：独立配置

### 11.2 后端

独立市场后端应有独立 Dockerfile 或复用后端基础镜像的独立启动构建。

例如：

- 容器名：`plugin-market-backend`
- SQLite 文件挂载目录：`/app/data`

### 11.3 Compose

建议新增独立 compose 文件，例如：

- `docker-compose.plugin-market.yml`

用于：

- 启动市场前端
- 启动市场后端
- 挂载 SQLite 数据目录

## 12. 测试策略

### 12.1 后端

至少覆盖：

1. SQLite 初始化与迁移测试
2. 插件同步幂等测试
3. 发布版本同步与下架测试
4. 插件列表查询测试
5. 插件详情查询测试
6. 同步 token 鉴权测试

### 12.2 前端

至少覆盖：

1. 列表页渲染测试
2. 详情页渲染测试
3. 中英文切换测试
4. 下载入口渲染测试
5. 空态与错误态测试

### 12.3 端到端

至少覆盖：

1. 独立市场前端启动
2. 独立市场后端启动
3. CMS 推送一条插件数据到市场服务
4. 市场前端列表可见该插件
5. 进入详情页可看到版本和下载链接
6. CMS 推送下架后，市场服务不再展示该版本或该插件

## 13. 风险与控制

### 风险 1：同步与查询模型脱节

控制方式：

- 同步 DTO 与查询 DTO 分离
- 版本同步以 `release_id` 为唯一键

### 风险 2：重复复用现有 CMS 逻辑导致边界重新耦合

控制方式：

- 新市场模块独立目录
- 不直接 import 现有 `plugin/service` 作为业务实现

### 风险 3：SQLite 数据被异常覆盖

控制方式：

- 所有同步接口采用 upsert
- 删除和下架分开处理
- Docker 挂载持久化数据目录

### 风险 4：前端仍被 CMS 运行时逻辑污染

控制方式：

- 独立前端应用独立 `app.tsx`
- 不复用 CMS 登录初始化和动态菜单机制

## 14. 推荐实施顺序

1. 新建 `feat/plugin-market-standalone` 分支
2. 搭建独立后端服务骨架与 SQLite 模型
3. 完成同步接口与公开查询接口
4. 搭建独立前端应用骨架
5. 迁移列表页与详情页
6. 完成 Docker 编排
7. 完成 CMS 推送协议联调
8. 做真实环境端到端测试

## 15. 结论

本方案将插件生态中心从“CMS 内部公开页面”升级为“独立部署、独立存储、独立同步的公开市场系统”：

- 前端独立
- 后端独立
- 数据库独立
- 领域模型独立
- 与 CMS 通过同步接口交互

同时保留当前 `master` 中的集成版插件生态中心，确保现有功能和未来拆分路径可以并行存在。
