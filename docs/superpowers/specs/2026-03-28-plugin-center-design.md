# 插件发布 - 插件中心重构设计

日期：2026-03-28

## 1. 背景与目标

`web-cms` 是一个包含多个业务模块的管理平台，插件发布只是其中一个模块。插件模块不应脱离现有 CMS 的菜单、角色、工作台和详情页交互模式单独裂出一套系统，而应在现有交互范式内完成重构。

当前插件中心存在的问题主要是：

- 项目、版本、流程的概念边界不清晰
- 首页信息过载，项目卡片承载了过多流程细节
- 审核、发布、项目管理的职责没有被明确拆开
- 项目详情页尚未形成统一的上下文承载中心
- 管理侧入口、页面结构与 `web-cms` 的现有菜单权限体系没有完全对齐

本次重构的目标是：

- 在不脱离 `web-cms` 现有交互模式的前提下，重构插件发布模块的信息架构
- 以“项目”为核心对象，以“版本”为操作主体，以“流程记录”为审计链路
- 根据角色和职责划分菜单、页面和默认视角
- 让提供方、审核者、发布者各自高效完成任务，同时共享统一的项目上下文

## 2. 设计原则

### 2.1 保持与 web-cms 一致

- 控制台仍然是全局首页
- 插件发布作为一个业务模块存在于 CMS 中
- 菜单和路由可见性由现有角色/菜单权限体系控制
- 不在插件模块内再发明一套独立的角色切换逻辑

### 2.2 项目是主对象

- 一个插件长期对应一个项目
- 项目承载基础资料与长期管理关系
- 版本是项目下的可操作单元
- 流程围绕版本展开

### 2.3 角色化入口，项目化详情

- 角色决定从哪个工作台进入模块
- 所有列表页最终都回到统一的项目详情页
- 项目详情页负责补齐上下文，而不是为不同角色维护三套详情页

### 2.4 管理系统优先高效

- 首页做轻量摘要，不堆叠流程细节
- 多版本并行以列表和筛选为主，不追求花哨展示
- 信息优先级明确：先看待办，再看上下文，再看审计记录

## 3. 角色定义

管理侧只讨论 3 类角色：

### 3.1 插件提供方

核心目标：

- 创建和维护项目
- 创建版本
- 上传版本资料
- 跟踪自己提交版本的流转状态
- 收到打回后补资料再提交
- 发起下架申请、归档旧版本

### 3.2 插件审核者

核心目标：

- 处理待审核版本
- 基于项目上下文理解当前版本改动
- 判断是否通过或打回
- 审核下架申请

### 3.3 插件发布者

核心目标：

- 处理已审核通过、待执行的版本
- 执行发布或执行下架
- 记录发布结果与发布链路

### 3.4 公开下载使用者

公开下载使用者不参与管理侧流程，不纳入本次管理端信息架构设计范围。

## 4. 核心业务对象

系统只保留 3 个核心对象：

### 4.1 项目 Project

项目是插件的长期主体，包含：

- 项目编码
- Git 仓库地址
- 中英文名称
- 中英文简短描述
- 中英文详细能力描述
- 负责人
- 项目整体状态

### 4.2 版本 Version

版本是流程和资料的承载主体，包含：

- 版本号
- 请求类型（首发 / 更新 / 下架）
- 当前状态
- 兼容信息（HCI / ACLI）
- 中英文变更说明
- 中英文下架原因
- 测试报告文件
- x86 插件包
- ARM 插件包
- 审核信息
- 发布信息

### 4.3 流程记录 Workflow / Audit Trail

流程记录用于保留完整可追踪审计链路，包括：

- 谁提交了版本
- 谁审核
- 审核意见
- 谁发布
- 何时发布
- 何时申请下架
- 谁批准下架
- 谁执行下架

## 5. 版本状态与流程关系

### 5.1 当前状态

版本对用户可见的当前状态应简洁明确：

- 草稿
- 筹备中
- 审核中
- 待发布
- 已发布
- 已下架
- 已归档

### 5.2 流程记录

版本同时保留完整流程历史，用于审计与链路追踪。

### 5.3 下架语义

下架同时具备两层语义：

- 对用户：体现为版本当前状态从“已发布”变为“已下架”
- 对系统：保留完整的下架申请、审核、执行链路

## 6. 菜单结构

插件发布模块在 `web-cms` 中保持为一个业务分组菜单：

- 插件发布
  - 项目管理
  - 审核工作台
  - 发布工作台

说明：

- 控制台仍然是系统首页
- 子菜单是否可见，由现有角色菜单权限决定
- 提供方、审核者、发布者只看到自己有权限访问的子菜单
- 多角色用户可同时看到多个子菜单

## 7. 页面组织方案

最终采用的页面组织方式为：

### 7.1 列表分开，详情统一

- 项目管理：项目列表页
- 审核工作台：待审核版本列表页
- 发布工作台：待发布版本列表页
- 统一项目详情页：所有列表都跳转到这里

该方案的优点：

- 角色入口清晰
- 详情上下文统一
- 项目仍然保持为核心对象
- 审核与发布不再演变成各自独立的“工单系统”

## 8. 页面树

### 8.1 项目管理

- 项目列表页
- 新建项目弹窗
- 编辑项目弹窗
- 项目详情页
- 创建新版本页

### 8.2 审核工作台

- 待审核版本列表页
- 点击记录后进入统一项目详情页，并自动定位目标版本

### 8.3 发布工作台

- 待发布版本列表页
- 点击记录后进入统一项目详情页，并自动定位目标版本

### 8.4 公共能力

- 下架申请弹窗
- 审核意见弹窗
- 发布确认弹窗
- 时间轴 / 审计记录组件
- 文件上传组件

## 9. 项目管理页设计

### 9.1 页面定位

项目管理页服务于插件提供方，是一个“轻项目首页”，主要回答：

- 我有哪些项目
- 每个项目最近卡在哪
- 我要不要进入某个项目继续处理

### 9.2 顶部区域

- 搜索：项目名称 / 编码 / 仓库 / 负责人
- 状态筛选：筹备中 / 审核中 / 已发布 / 已归档
- 兼容版本筛选：HCI / ACLI
- 视图切换：卡片 / 列表
- 新建项目按钮

### 9.3 项目卡片

卡片只展示轻量项目信息：

- 项目图标
- 中英文名称
- 中英文简短描述
- 最新版本号
- 最近一条流程摘要
- 项目整体状态标签

最近流程摘要只用纯文本，例如：

- `v1.2.0 审核中`
- `v1.3.0 待补资料`
- `v1.1.0 已发布`

首页不展示多个版本的并行流程，以避免卡片再次膨胀。

### 9.4 列表视图

字段建议：

- 项目名称
- 项目编码
- 简短描述
- 负责人
- 最新版本
- 最近流程摘要
- 项目状态
- 操作

### 9.5 首页操作原则

项目管理首页只做轻量入口操作：

- 新建项目
- 编辑项目
- 进入项目
- 快速创建新版本

不在首页处理完整流程。

## 10. 审核工作台设计

### 10.1 页面定位

审核工作台服务于审核者，目标是高效处理待审核版本，而不是浏览项目。

### 10.2 顶部摘要区

- 待审核版本数
- 待审核下架申请数
- 今日已审核数
- 最近打回数

### 10.3 筛选区

- 搜索：项目名 / 版本号 / 提交人
- 类型筛选：首发 / 更新 / 下架
- 状态筛选：待审核 / 已打回 / 已通过
- 时间筛选：最近 7 天 / 30 天
- 是否只看分配给我

### 10.4 列表主体

默认展示：

- 待审核
- 我审核过的

列表字段建议：

- 项目名称
- 版本号
- 请求类型
- 当前状态
- 提交人
- 提交时间
- 最近更新时间
- 是否待我处理

### 10.5 列表交互

- 列表页不直接执行审核操作
- 点击整行后，进入项目详情页
- 自动定位到对应待审核版本
- 默认打开适合审核的面板

## 11. 发布工作台设计

### 11.1 页面定位

发布工作台服务于发布者，目标是处理已通过审核、待执行发布或下架的版本。

### 11.2 顶部摘要区

- 待发布版本数
- 待执行下架数
- 今日已发布数
- 今日已下架数

### 11.3 筛选区

- 搜索：项目名 / 版本号 / 发布人
- 类型筛选：首发 / 更新 / 下架
- 状态筛选：待发布 / 已发布 / 已下架
- 是否只看分配给我
- 时间筛选

### 11.4 列表主体

默认展示：

- 待发布
- 我发布过的

列表字段建议：

- 项目名称
- 版本号
- 请求类型
- 当前状态
- 审核结论
- 审核人
- 审核完成时间
- 是否待我处理

### 11.5 列表交互

- 列表页不直接执行发布
- 点击整行后进入项目详情页
- 自动定位到对应待发布版本
- 默认打开适合发布的面板

## 12. 项目详情页设计

项目详情页是统一上下文承载页，是整个插件发布模块的中心页。

### 12.1 页面整体结构

- 左侧：项目基础信息
- 中间：版本列表
- 右侧顶部：流程区
- 右侧下方：Tabs

### 12.2 左侧项目基础信息

包含：

- 项目图标
- 中英文项目名称
- 项目编码
- 仓库地址
- 负责人
- 中英文详细描述
- 项目整体状态
- 编辑项目按钮（仅提供方）

### 12.3 中间版本列表

中间版本列表采用高效后台式设计：

- 固定宽度
- 支持滚动
- 进行中版本优先
- 支持筛选：
  - 全部
  - 进行中
  - 已发布
  - 已下架 / 已归档

每个版本项最少显示：

- 版本号
- 当前状态
- 请求类型
- 更新时间
- 是否待我处理

### 12.4 右侧顶部流程区

流程区固定在顶部，始终可见。

默认流程：

- 提交资料
- 审核
- 发布

下架流程：

- 下架申请
- 下架审核
- 执行下架

流程区展示：

- 当前节点
- 当前状态
- 当前处理角色 / 处理人
- 当前主操作按钮

### 12.5 右侧 Tabs

固定 4 个 Tab：

- 版本概览
- 文件资料
- 审核与发布
- 时间轴

### 12.6 不同入口的默认落点

#### 从项目管理进入

- 默认选中最近活跃版本
- 默认打开“版本概览”

#### 从审核工作台进入

- 默认选中目标待审核版本
- 默认打开“审核与发布”

#### 从发布工作台进入

- 默认选中目标待发布版本
- 默认打开“审核与发布”

## 13. 项目详情页 Tabs 设计

### 13.1 版本概览

用于快速理解这个版本是什么、改了什么、适配什么。

字段建议：

- 版本号
- 请求类型
- 当前状态
- 提交人
- 提交时间
- 适配 HCI 版本
- 适配 ACLI 版本
- 中英文变更说明
- 若为下架流程，则显示中英文下架原因

该 Tab 不承载主操作按钮。

### 13.2 文件资料

用于确认资料是否齐全，以及下载查看。

展示文件：

- 测试报告
- x86 插件包
- ARM 插件包

每张文件卡显示：

- 文件类型
- 文件名
- 文件大小
- 上传时间
- 下载按钮

对于仍可编辑的版本，提供方可以执行上传或替换文件。

### 13.3 审核与发布

用于展示该版本当前的提交、审核、发布处理信息。

建议分为 3 个区块：

- 提交信息
- 审核信息
- 发布信息

其中包括：

- 提交人
- 提交时间
- 审核人
- 审核状态
- 审核意见
- AI 扫描结果摘要
- 发布者
- 发布状态
- 全面扫描结果
- 发布时间
- 下架执行信息

不同角色的主要操作挂在此区域或流程区附近。

### 13.4 时间轴

用于完整保留流程链路和审计信息。

每条记录包含：

- 时间
- 操作人
- 动作
- 状态变化
- 备注 / 审核意见 / 下架原因

推荐使用正序时间轴，便于理解整个版本从创建到结束的流转过程。

## 14. 创建新版本策略

创建新版本采用“先创建版本壳，再在项目详情里补齐资料”的方式。

### 14.1 最小创建动作

点击“创建新版本”后，只填写最小必要信息：

- 版本号
- 请求类型（首发 / 更新）
- 兼容版本范围

### 14.2 创建后状态

版本创建成功后：

- 新版本出现在左侧版本列表
- 当前状态为“筹备中”
- 提供方随后在项目详情中继续补充资料

### 14.3 后续完善

在项目详情页中继续补：

- 中英文变更说明
- 测试报告
- x86 / ARM 包
- 其他资料

资料齐全后，再执行“提交资料”，使流程进入审核。

## 15. 操作策略

### 15.1 弹窗操作

适合短输入、轻确认：

- 新建项目
- 编辑项目基础信息
- 审核通过确认
- 审核打回（必须填写意见）
- 发布确认
- 下架申请
- 归档确认

### 15.2 独立页面操作

适合复杂资料和较长填写流程：

- 创建新版本页
- 复杂版本资料编辑页（如后续拆分）

### 15.3 页内直接操作

适合低风险、无输入操作：

- 切换版本
- 下载文件
- 复制仓库地址

## 16. 角色操作边界

### 16.1 提供方

允许：

- 创建项目
- 编辑项目
- 创建新版本
- 编辑当前可编辑版本资料
- 提交资料
- 重新提交
- 发起下架申请
- 归档已发布版本

### 16.2 审核者

允许：

- 查看项目与版本资料
- 审核通过
- 打回并填写意见
- 审核下架申请

不允许：

- 编辑项目基础信息
- 直接执行发布

### 16.3 发布者

允许：

- 查看项目与版本资料
- 执行发布
- 执行下架
- 填写发布结果 / 扫描结果摘要

不允许：

- 编辑项目基础信息
- 执行审核决策

## 17. 实施建议顺序

建议按以下顺序推进实施：

1. 先重构插件发布模块的菜单与页面树
2. 建立统一项目详情页骨架
3. 重构项目管理页
4. 重构审核工作台和发布工作台
5. 补齐创建新版本、下架、审核、发布等关键交互
6. 最后补全视觉细节与角色体验优化

## 18. 暂不处理范围

本次 spec 暂不覆盖：

- 公开插件市场的视觉和交互
- 插件购买、评分、收藏等市场功能
- 与外部通知系统的具体联动方式
- 超出当前角色模型之外的更多角色扩展
